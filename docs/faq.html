For each File — Frequently Asked Questions

This document attempts to answer frequently asked questions about the For each File package.

Questions and Answers

  1. What are the advantages of using For each File instead of find -exec?

    There is one advantage, which can be observed under the following conditions. If find -exec is used to invoke an intermediate shell: $ find . -exec sh -c '...' \;, then it will expend more system resources than For each File. This is because, for each file it handles, find -exec will fork a child process (the intermediate shell) and thereby cause the operating system to perform a context switch. In contrast, For each File will invoke a GNU BASH function that is already loaded into memory and thus does not cause a context switch. This behavior reduces the number of process forks and thereby expends less system resources as compared to find -exec under said conditions.

  2. Why is the read built-in command in my script being ignored?

    By default, the read built-in command attempts to read user input from the standard input stream. However, the standard input stream may be in use if this tool is invoked with the --pipe option.

    If your intention is to read input from the user, a solution is to have the read built-in command read directly from the user's terminal rather than the standard input stream: read < /dev/tty

  3. Why does the tool crash when a large number of targets are specified on the command-line?

    The GNU BASH shell employs a buffer, of finite capacity, in which it stores command-line arguments. When this buffer is filled beyond its capacity by a user's command, the user's command may crash. A solution is to pipe the large list of targets to the tool whilst invoking it with the --pipe option.

    Return to Question Listing.

  4. Why are some targets given on the command-line handled improperly?

    The names of these targets may contain characters that are being misinterpreted by the tool or the shell. A solution is to feed a list of the target names to the tool whilst invoking it with the --pipe option. This ensures that target names bypass the scrutiny of both the shell and tool, thus avoiding misinterpretation.
